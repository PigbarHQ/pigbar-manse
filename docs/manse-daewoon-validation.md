# Pigbar Manse 대운 계산 기준 검증

작성일: 2026-07-03

이 문서는 기능 추가 없이 현재 Pigbar Manse의 대운 계산 기준을 검토하기 위한 문서다. 기준 코드는 `src/lib/manse/daewoon.ts`, `src/lib/manse/currentLuck.ts`, `src/lib/manse/index.ts`이며, 사주 원국 계산은 `@fullstackfamily/manseryeok`를 사용한다.

## 1. 현재 로직 설명

### 1.1 대운 계산 흐름

현재 대운 계산은 다음 순서로 진행된다.

1. 입력 생년월일을 양력 기준으로 정규화한다.
2. 출생지 경도 기준 지역시 보정을 적용한다.
3. `@fullstackfamily/manseryeok`로 년주, 월주, 일주, 시주를 계산한다.
4. 년주의 음양과 성별로 순행/역행을 결정한다.
5. 순행이면 출생 이후의 다음 절기, 역행이면 출생 이전의 이전 절기를 찾는다.
6. 출생 시각과 대상 절기 시각의 차이를 대운 시작 나이로 환산한다.
7. 월주를 기준으로 순행/역행 방향에 따라 첫 대운 간지를 만든다.
8. 첫 대운 시작 시각부터 10년 단위 대운 타임라인을 만든다.
9. `currentDateTime >= cycle.startDateTime && currentDateTime < cycle.endDateTime` 조건으로 현재 대운을 선택한다.

### 1.2 순행/역행 결정 기준

현재 코드는 년간의 음양과 성별을 기준으로 한다.

```text
양년 남자 = 순행
음년 여자 = 순행
음년 남자 = 역행
양년 여자 = 역행
```

코드 기준:

```ts
const isYangYear = yearPillar.stem.yinYang === "yang";
if ((isYangYear && gender === "male") || (!isYangYear && gender === "female")) {
  return "forward";
}
return "backward";
```

이는 한국 만세력에서 흔히 쓰는 `양남음녀 순행, 음남양녀 역행` 기준과 일치한다.

### 1.3 첫 대운 간지 결정 기준

현재 코드는 월주를 60갑자 인덱스로 변환한 뒤 다음 간지 또는 이전 간지를 첫 대운으로 사용한다.

```text
순행: 월주 + 1
역행: 월주 - 1
```

예: 월주가 `신미(辛未)`이고 순행이면 다음 간지인 `임신(壬申)`이 첫 대운이다.

이 기준은 일반적인 한국 만세력 표시와 일치한다. 월주는 대운의 기준점이며, 첫 실제 대운은 순행/역행에 따라 월주의 다음/이전 간지부터 시작한다.

### 1.4 대운 시작 나이 계산식

현재 계산식은 코드 주석상 다음 기준이다.

```text
3일 = 1년
1일 = 4개월
1시간 = 5일
```

구현은 다음 형태다.

```ts
const diffDays = abs(targetTerm.diff(birth, "days").days);
const totalMonths = Math.round(diffDays * 4);
const years = Math.floor(totalMonths / 12);
const months = totalMonths % 12;
const age = years + (months > 0 ? 1 : 0);
const days = Math.round(diffDays);
```

즉, 절기까지의 일수 차이를 월수로 환산해 `years/months`를 만든다. 다만 `days`는 `years/months`로 환산하고 남은 잔여 일이 아니라, 원래 절기 차이 일수를 반올림한 값이다. 따라서 현재 출력의 `2년 11개월 9일`은 엄밀한 기간 분해라기보다 `2년 11개월경, 절기 차이 약 9일`에 가깝다.

### 1.5 절기 시각 데이터

`@fullstackfamily/manseryeok` README 기준:

- 월주는 절기 기준으로 계산한다.
- 절기 시각 조회 API가 있다.
- 현재 내장 정밀 절기 시각 지원 연도는 `2020-2030`이다.

Pigbar Manse는 `getSolarTermsByYear(year)`를 호출한다. 해당 연도가 지원되지 않으면 코드 내부의 근사 절입일을 사용한다.

현재 근사 절입일:

```text
1/6, 2/4, 3/6, 4/5, 5/6, 6/6,
7/7, 8/8, 9/8, 10/8, 11/7, 12/7
```

시간은 모두 `00:00 Asia/Seoul`로 처리한다. 따라서 1974년 대운 시작 시점은 정밀 절기 시각이 아니라 근사값이다.

## 2. 1974-07-30 03:50 서울 남성 검증

입력:

```json
{
  "gender": "male",
  "birthDate": "1974-07-30",
  "calendarType": "solar",
  "birthTime": "03:50",
  "birthPlace": "서울특별시, 대한민국",
  "useLocalMeanTime": true,
  "currentDateTime": "2026-07-02T12:30:00+09:00"
}
```

지역시 보정:

```text
표준 경도: 135
서울 경도: 126.978
보정값: -32.088분
원 입력: 1974-07-30 03:50
보정 후: 1974-07-30 03:17:54.720+09:00
```

원국:

```text
년주: 갑인(甲寅)
월주: 신미(辛未)
일주: 임신(壬申)
시주: 임인(壬寅)
```

대운:

```text
대운 방향: 순행
대운 시작: 2년 11개월 9일경
표시상 대운수: 약 3세
첫 대운: 임신(壬申)
현재 대운(2026-07-02): 병자(丙子)
다음 대운: 정축(丁丑), 2027-07-09 시작
```

현재 생성되는 주요 대운 타임라인:

| index | 대운 | 시작 | 종료 |
|---:|---|---|---|
| 1 | 임신 | 1977-07-09T03:17:54.720+09:00 | 1987-07-09T03:17:54.720+10:00 |
| 2 | 계유 | 1987-07-09T03:17:54.720+10:00 | 1997-07-09T03:17:54.720+09:00 |
| 3 | 갑술 | 1997-07-09T03:17:54.720+09:00 | 2007-07-09T03:17:54.720+09:00 |
| 4 | 을해 | 2007-07-09T03:17:54.720+09:00 | 2017-07-09T03:17:54.720+09:00 |
| 5 | 병자 | 2017-07-09T03:17:54.720+09:00 | 2027-07-09T03:17:54.720+09:00 |
| 6 | 정축 | 2027-07-09T03:17:54.720+09:00 | 2037-07-09T03:17:54.720+09:00 |

## 3. 다른 만세력의 “대운수 3(신미)” 해석

다른 만세력에서 보이는 `대운수 3(신미)`는 보통 다음처럼 해석하는 것이 자연스럽다.

```text
대운수 3 = 대운이 약 3세부터 시작한다.
신미 = 출생 월주, 즉 대운 전개의 기준이 되는 월주다.
```

즉 `신미`가 첫 대운이라는 뜻이라기보다, 대운을 밀고 나가는 기준 월주를 함께 표시한 것일 가능성이 높다.

Pigbar Manse의 현재 기준에서는:

```text
월주: 신미
대운 방향: 순행
첫 대운: 임신
```

따라서 `대운수 3(신미)`라는 화면과 `첫 대운 임신`은 서로 모순이 아니다. 다만 서비스 UI에서는 `기준 월주: 신미`, `첫 대운: 임신`, `대운 시작: 약 3세`를 분리해 보여줘야 오해가 적다.

## 4. 검증 결과

### 일치하는 부분

- 월주 `신미`는 `@fullstackfamily/manseryeok` 기준 결과와 일치한다.
- 갑인년 남성이므로 순행으로 판단하는 것은 일반 한국 만세력 기준과 일치한다.
- 순행에서 월주 `신미`의 다음 간지 `임신`을 첫 대운으로 쓰는 것은 일반 기준과 일치한다.
- 대운 시작을 약 3세로 보는 결과는 다른 만세력의 `대운수 3` 표시와 큰 틀에서 일치한다.
- 현재 대운 판정은 실제 시작/종료 시각 범위로 수행되고 있어, 2026년에 2027년 시작 대운이 선택되는 이전 오류는 해결되어 있다.

### 불일치 또는 불확실한 부분

- 1974년의 정밀 절기 시각이 없다. 현재는 8월 8일 00:00 근사 절입값을 사용한다.
- `ageDays`가 잔여 일수가 아니라 절기까지의 원래 차이 일수를 반올림한 값이다. 따라서 `2년 11개월 9일`이라는 표현은 엄밀한 기간 표현으로 보기 어렵다.
- `1시간 = 5일`이라는 주석은 있으나, 현재 구현은 `diffDays * 4`를 반올림하여 월수로 환산한다. 시간 단위 환산을 별도 일수/시수로 세밀하게 분해하지 않는다.
- 1987년 한국 서머타임 때문에 Luxon 결과에 `+10:00`이 나타난다. 만세력 대운 경계 표시에서 역사적 KST/DST를 반영할지, 고정 `+09:00`으로 표시할지 정책 결정이 필요하다.

## 5. 수정 필요 여부

현재 1974년 케이스의 핵심 대운 기준은 다음과 같이 판단한다.

```text
월주 신미 -> 순행 -> 첫 대운 임신 -> 대운수 약 3세
```

이 큰 구조는 한국 만세력 기준과 맞는다. 따라서 첫 대운 간지나 순행/역행 로직을 바꿀 필요는 없다.

다만 정식 서비스 안정화 관점에서는 다음 수정이 필요하다.

1. 1900-2050 범위의 정밀 절기 시각 확보
2. 대운 시작 기간 표현의 의미 정리
3. `ageDays` 필드 명확화 또는 잔여 일수 기반으로 재설계
4. 대운 경계 표시에서 KST 고정 여부 결정

## 6. 정밀 절기 데이터 해결 후보

### 후보 A: 정밀 절기 테이블 내장

1900-2050 범위의 절기 시각을 검증된 데이터로 확보해 JSON/TS 테이블로 내장한다.

장점:

- 런타임 의존성이 적다.
- 테스트 고정이 쉽다.
- 배포 후 결과가 안정적이다.

단점:

- 데이터 출처와 라이선스 검토가 필요하다.
- 데이터 갱신/검증 파이프라인을 만들어야 한다.

### 후보 B: 천문 계산 라이브러리 도입

태양 황경 기준으로 24절기 시각을 직접 계산하는 라이브러리를 사용한다.

장점:

- 긴 연도 범위로 확장 가능하다.
- 별도 테이블 유지 부담이 줄어든다.

단점:

- 한국 표준시, 역법 보정, 오차 검증이 필요하다.
- 기존 만세력 서비스와 분 단위 차이가 날 수 있다.

### 후보 C: `@fullstackfamily/manseryeok`에 절기 데이터 확장 기여

현재 라이브러리가 이미 2020-2030 절기 시각 API를 제공하므로, 동일 구조로 1900-2050 데이터를 확장한다.

장점:

- 현재 의존 구조를 유지할 수 있다.
- 월주 계산과 대운 계산의 절기 기준을 한 라이브러리로 맞출 수 있다.

단점:

- 업스트림 반영 또는 포크 유지가 필요하다.

## 결론

Pigbar Manse의 현재 대운 계산은 방향 결정과 첫 대운 간지 결정 면에서는 한국 만세력 관례와 일치한다. 1974-07-30 03:50 서울 남성 케이스도 `월주 신미`, `순행`, `첫 대운 임신`, `대운수 약 3세`로 해석하는 것이 타당하다.

다만 1974년 정밀 절기 시각이 없기 때문에 대운 시작 시점은 `estimated`로 유지해야 한다. 정식 서비스 수준으로 올리려면 과거 절기 시각 데이터를 확보하고, 대운 시작 기간 필드의 의미를 더 엄밀하게 정리해야 한다.
